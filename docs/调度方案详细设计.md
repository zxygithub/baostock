## 1. 设计目标

### 1.1 核心诉求

| 目标 | 说明 |
|------|------|
| **可配置** | 基于数据库表 / 数据种类进行任务配置，无需修改代码 |
| **可量化** | 支持时间范围、拉取数据量、请求次数等维度的精确控制 |
| **可复用** | 程序每次启动时自动读取调度配置，按配置执行拉取 |
| **可观测** | 每次调度运行有完整的执行记录和统计 |
| **可恢复** | 中断后可从断点恢复，不重复拉取已完成任务 |

### 1.2 设计原则

- **向后兼容**：现有 `config.yaml` 和下载脚本无需修改即可继续工作
- **渐进式接入**：调度器作为可选层，不强制替代现有下载流程
- **最小侵入**：复用现有 Downloader 体系，不重写下载逻辑
- **单一职责**：调度器只负责任务编排，不关心具体下载实现

## 2. 现状分析

### 2.1 当前下载流程

```
start.sh full/update
    ↓
scripts/download_all.py / update_daily.py
    ↓
按固定顺序调用各 Downloader
    ↓
每个 Downloader 独立管理自己的下载逻辑
```

### 2.2 存在的问题

| 问题 | 影响 |
|------|------|
| 下载顺序硬编码 | 调整顺序需修改 Python 代码 |
| 请求配额全局共享 | 无法为不同数据种类分配不同配额 |
| 无任务级状态追踪 | 不知道哪些表已拉完、哪些失败 |
| 全量/增量两套脚本 | 维护成本高，逻辑有重复 |
| 配置分散 | `config.yaml` 控制开关和日期，代码中控制顺序和参数 |

### 2.3 现有可复用组件

| 组件 | 可复用点 |
|------|----------|
| `BaseDownloader` | 登录、重试、checkpoint、请求计数 |
| `config_loader.py` | YAML 配置加载机制 |
| `db_manager.py` | SQLite 连接管理、表结构定义 |
| 各具体 Downloader | 实际下载逻辑（无需修改） |
| `request_count` 表 | 已有的 API 请求计数基础 |

## 3. 总体架构

### 3.1 架构图

```
┌─────────────────────────────────────────────────────────┐
│                    程序启动 (start.sh)                    │
└────────────────────────┬────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│              SchedulerOrchestrator (调度编排器)            │
│                                                         │
│  1. 加载 scheduler.yaml → SchedulerConfig                │
│  2. 初始化 scheduler_tasks / scheduler_runs 表           │
│  3. 计算当日可执行任务列表                                 │
│  4. 按 priority 排序，依次执行                            │
│  5. 更新任务状态，写入运行记录                             │
└────────┬──────────────────────────────┬─────────────────┘
         │                              │
         ▼                              ▼
┌──────────────────┐        ┌──────────────────────────┐
│  TaskExecutor    │        │  QuotaManager (配额管理)   │
│  (任务执行器)     │        │                          │
│                  │        │  - 每日总配额检查           │
│  - 解析 task_def │        │  - 按任务分配配额           │
│  - 实例化对应    │        │  - 实时消耗追踪             │
│    Downloader    │        │  - 配额不足时跳过/降级      │
│  - 执行下载      │        └──────────────────────────┘
│  - 回写状态      │
└──────────────────┘
```

### 3.2 模块职责

| 模块 | 文件 | 职责 |
|------|------|------|
| `SchedulerConfig` | `src/scheduler/config.py` | 加载并解析 `scheduler.yaml` |
| `SchedulerOrchestrator` | `src/scheduler/orchestrator.py` | 任务编排：加载配置 → 筛选任务 → 排序 → 执行 |
| `TaskExecutor` | `src/scheduler/executor.py` | 将任务定义映射到具体 Downloader 并执行 |
| `QuotaManager` | `src/scheduler/quota.py` | API 请求配额的分配、检查、消耗追踪 |
| `scheduler.yaml` | 项目根目录 | 用户可编辑的调度配置文件 |

```
       │                              │

┌──────────────┐           ┌─────────────────────┐

└──────────────┘           └──────────┬──────────┘
                                      │
                                      ▼
                            ┌─────────────────────┐
                            │ SchedulerOrchestrator│

                            │   TaskExecutor       │
                            │       ↓              │
                            │   QuotaManager       │
                            └──────────┬──────────┘
```

## 4. 数据库设计

### 4.1 新增表清单

| 序号 | 表名 | 说明 |
|------|------|------|
| 1 | `scheduler_tasks` | 调度任务定义（配置持久化） |
| 2 | `scheduler_runs` | 调度运行记录（每次执行的历史） |
| 3 | `scheduler_task_runs` | 任务级执行记录（每个任务的执行结果） |
| 4 | `quota_allocation` | 配额分配与消耗记录 |

### 4.2 表结构

#### 4.2.1 `scheduler_tasks` — 调度任务定义

```sql
CREATE TABLE IF NOT EXISTS scheduler_tasks (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    task_name       TEXT    NOT NULL UNIQUE,  -- 任务唯一标识，如 "kline_daily"
    data_category   TEXT    NOT NULL,         -- 数据大类：kline / financial / macro / report / dividend / index / component / meta
    data_subcategory TEXT,                    -- 数据子类：daily / weekly / monthly / profit / operation 等
    target_tables   TEXT    NOT NULL,         -- 目标表名列表，JSON 数组：["all_stock_daily"]
    enabled         INTEGER NOT NULL DEFAULT 1, -- 是否启用：0=禁用, 1=启用
    priority        INTEGER NOT NULL DEFAULT 10, -- 执行优先级，数字越小越先执行
    schedule_type   TEXT    NOT NULL DEFAULT 'on_startup', -- 触发类型：on_startup / daily / weekly / monthly / manual
    schedule_config TEXT,                     -- 调度参数，JSON 对象（见下方说明）
    download_params TEXT,                     -- 下载参数，JSON 对象（传递给 Downloader 的参数）
    quota_config    TEXT,                     -- 配额配置，JSON 对象（见下方说明）
    retry_policy    TEXT,                     -- 重试策略，JSON 对象
    created_at      TEXT    NOT NULL,         -- 创建时间
    updated_at      TEXT    NOT NULL          -- 最后更新时间
);
```

**`schedule_config` JSON 结构：**

```json
{
  "time_window": {
    "start": "01:00",
    "end": "06:00"
  },
  "weekdays": [1, 2, 3, 4, 5],
  "trading_day_only": true,
  "max_duration_minutes": 120,
  "date_range": {
    "start_date": "1990-12-19",
    "end_date": "auto",
    "end_offset_days": -1
  }
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `time_window.start` | string | 允许执行开始时间（HH:MM），超出窗口不执行 |
| `time_window.end` | string | 允许执行结束时间 |
| `weekdays` | array | 允许执行的星期（1=周一，7=周日），空表示不限 |
| `trading_day_only` | bool | 是否仅交易日执行 |
| `max_duration_minutes` | int | 最大执行时长（分钟），超时强制终止 |
| `date_range.start_date` | string | 数据起始日期 |
| `date_range.end_date` | string | 数据结束日期，`"auto"` 表示自动计算 |
| `date_range.end_offset_days` | int | 相对于今天的偏移天数，-1 表示昨天 |

**`download_params` JSON 结构（示例）：**

```json
{
  "batch_size": 200,
  "batch_sleep": 2,
  "adjustflags": [1, 2, 3],
  "stock_filter": "type=1 AND status=1"
}
```

**`quota_config` JSON 结构：**

```json
{
  "max_requests": 30000,
  "max_rows": null,
  "max_duration_minutes": 60,
  "share_of_daily_limit": 0.3
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `max_requests` | int | 本任务最大 API 请求次数 |
| `max_rows` | int | 本任务最大拉取行数（null 表示不限） |
| `max_duration_minutes` | int | 本任务最大执行时长 |
| `share_of_daily_limit` | float | 占每日总配额的比例（0.3 = 30%） |

**`retry_policy` JSON 结构：**

```json
{
  "max_retries": 3,
  "retry_on_error_codes": ["10002002", "10002003"],
  "backoff_base": 2,
  "skip_on_quota_exceeded": true
}
```

#### 4.2.2 `scheduler_runs` — 调度运行记录

```sql
CREATE TABLE IF NOT EXISTS scheduler_runs (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id          TEXT    NOT NULL UNIQUE,  -- 运行唯一标识 (UUID)
    trigger_type    TEXT    NOT NULL,         -- 触发方式：startup / cron / manual
    started_at      TEXT    NOT NULL,         -- 开始时间
    finished_at     TEXT,                     -- 结束时间
    status          TEXT    NOT NULL DEFAULT 'running', -- running / completed / failed / interrupted
    total_tasks     INTEGER NOT NULL DEFAULT 0, -- 计划执行任务数
    completed_tasks INTEGER NOT NULL DEFAULT 0, -- 已完成任务数
    failed_tasks    INTEGER NOT NULL DEFAULT 0, -- 失败任务数
    skipped_tasks   INTEGER NOT NULL DEFAULT 0, -- 跳过任务数
    total_requests  INTEGER NOT NULL DEFAULT 0, -- 总 API 请求数
    total_rows      INTEGER NOT NULL DEFAULT 0, -- 总拉取行数
    error_message   TEXT,                     -- 错误信息（如有）
    config_snapshot TEXT                      -- 本次运行的配置快照 (JSON)
);
```

#### 4.2.3 `scheduler_task_runs` — 任务级执行记录

```sql
CREATE TABLE IF NOT EXISTS scheduler_task_runs (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id          TEXT    NOT NULL,         -- 关联 scheduler_runs.run_id
    task_name       TEXT    NOT NULL,         -- 关联 scheduler_tasks.task_name
    started_at      TEXT    NOT NULL,         -- 任务开始时间
    finished_at     TEXT,                     -- 任务结束时间
    status          TEXT    NOT NULL DEFAULT 'running', -- running / completed / failed / skipped / interrupted
    requests_used   INTEGER NOT NULL DEFAULT 0, -- 本次执行 API 请求数
    rows_fetched    INTEGER NOT NULL DEFAULT 0, -- 本次执行拉取行数
    tables_updated  TEXT,                     -- 更新的表名列表 (JSON 数组)
    error_message   TEXT,                     -- 错误信息
    checkpoint      TEXT,                     -- checkpoint 数据 (JSON)
    duration_seconds REAL                     -- 执行耗时（秒）
);

CREATE INDEX idx_task_runs_run_id ON scheduler_task_runs(run_id);
CREATE INDEX idx_task_runs_task_name ON scheduler_task_runs(task_name);
```

#### 4.2.4 `quota_allocation` — 配额分配与消耗

```sql
CREATE TABLE IF NOT EXISTS quota_allocation (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    date            TEXT    NOT NULL,         -- 日期 (YYYY-MM-DD)
    task_name       TEXT    NOT NULL,         -- 任务名称
    allocated       INTEGER NOT NULL DEFAULT 0, -- 分配的请求配额
    consumed        INTEGER NOT NULL DEFAULT 0, -- 已消耗的请求数
    remaining       INTEGER NOT NULL DEFAULT 0, -- 剩余配额
    updated_at      TEXT,                     -- 最后更新时间
    UNIQUE(date, task_name)
);

CREATE INDEX idx_quota_date ON quota_allocation(date);
```

### 4.3 与现有表的关系

```
现有表                          新增调度表
┌──────────────┐               ┌──────────────────┐
│ request_count│◄──────────────│ quota_allocation │

└──────────────┘               └──────────────────┘

                               ┌──────────────────┐

                               ┌────────▼─────────┐

                               │ (运行记录)        │
                               └────────┬─────────┘
                                        │
                               ┌────────▼─────────┐
                               │scheduler_task_runs│
                               │ (任务执行记录)    │
                               └──────────────────┘
```

## 5. 调度配置模型

### 5.1 `scheduler.yaml` 文件格式

```yaml

# BaoStock 数据拉取任务调度配置
# ============================================================================

# 全局设置
global:
  # 每日 API 请求总上限（覆盖 config.yaml 中的 api.daily_request_limit）
  daily_request_limit: 49000
  # 是否启用调度器（false 时回退到原有脚本执行方式）
  enabled: true
  # 默认重试策略
  default_retry:
    max_retries: 3
    backoff_base: 2

# 任务定义列表（按 priority 从小到大排序执行）
tasks:
  # -----------------------------------------------------------------------
  # 元数据任务（最高优先级，其他任务依赖股票列表）
  # -----------------------------------------------------------------------
  - name: meta_basic
    category: meta
    subcategory: basic
    tables: ["stock_basic", "trade_dates", "stock_industry"]
    enabled: true
    priority: 1
    schedule:
      type: on_startup
      trading_day_only: false
    download_params:
      full_refresh: true
    quota:
      max_requests: 50
      share_of_daily_limit: 0.001

  # -----------------------------------------------------------------------
  # 指数成分股（每周一全量更新）
  # -----------------------------------------------------------------------
  - name: index_components
    category: component

    enabled: true
    priority: 2
    schedule:
      type: weekly
      weekdays: [1]  # 周一

    quota:
      max_requests: 100
      share_of_daily_limit: 0.002

  # -----------------------------------------------------------------------
  # 宏观经济数据（数据量小，全量拉取）
  # -----------------------------------------------------------------------
```

### 5.2 配置字段说明

#### 任务级字段

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `name` | string | 是 | - | 任务唯一标识 |

| `subcategory` | string | 否 | null | 数据子类 |
| `tables` | array | 是 | - | 目标数据库表名列表 |
| `enabled` | bool | 否 | true | 是否启用 |
| `priority` | int | 否 | 10 | 执行优先级（小→大） |

| `download_params` | object | 否 | {} | 传递给 Downloader 的参数 |

| `trading_day_only` | bool | 是否仅交易日执行 |
| `time_window` | object | 执行时间窗口 `{start: "01:00", end: "06:00"}` |
| `max_duration_minutes` | int | 最大执行时长 |

## 6. 核心模块设计

### 6.2 `QuotaManager` — 配额管理器

**文件**: `src/scheduler/quota.py`

```python
class QuotaManager:
    """管理 API 请求配额的分配、检查和消耗。"""

    def __init__(self, db_path: str, global_limit: int):
        self.db_path = db_path
        self.global_limit = global_limit
        self._allocations: dict[str, int] = {}  # task_name -> allocated
        self._consumed: dict[str, int] = {}     # task_name -> consumed

    def allocate(self, tasks: list[TaskDefinition]) -> None:
        """为任务列表分配配额。支持两种模式：
        1. 固定配额：使用 task.quota.max_requests
        2. 比例分配：使用 task.quota.share_of_daily_limit
        比例不足 1.0 时，剩余配额按 max_requests 分配。
        """

    def can_execute(self, task_name: str) -> bool:
        """检查任务是否还有可用配额。"""

    def consume(self, task_name: str, count: int = 1) -> bool:
        """消耗配额。返回 True 表示成功，False 表示配额不足。"""

    def get_remaining(self, task_name: str) -> int:
        """获取任务剩余配额。"""

    def get_global_remaining(self) -> int:
        """获取全局剩余配额。"""

    def persist(self, date: str) -> None:
        """将配额状态写入 quota_allocation 表。"""
```

**配额分配算法**：

```
输入：tasks[], global_limit

1. 收集所有任务的 share_of_daily_limit 和 max_requests
2. 计算比例配额总和 ratio_sum = Σ(share_of_daily_limit)
3. 如果 ratio_sum > 1.0，按比例缩减
4. 对每个任务：
   a. 如果有 share_of_daily_limit → allocated = global_limit * share
   b. 如果有 max_requests → allocated = min(max_requests, 剩余配额)
   c. 如果两者都有 → allocated = min(两者计算结果)
5. 写入 quota_allocation 表
```

```python
class TaskExecutor:
    """将任务定义映射到具体 Downloader 并执行下载。"""

    # 任务类别到 Downloader 的映射
    CATEGORY_MAP = {
        "meta": MetaDownloader,
        "macro": MacroDownloader,
        "component": ComponentDownloader,
        "index": IndexDownloader,
        "kline": KlineDownloader,
        "financial": FinancialDownloader,
        "report": ReportDownloader,
        "dividend": DividendDownloader,
    }

    def __init__(self, db_path: str, logger: logging.Logger):
        self.db_path = db_path
        self.logger = logger

    def execute(self, task: TaskDefinition, quota_mgr: QuotaManager) -> TaskRunResult:
        """执行单个任务。

        流程：
        1. 检查配额是否充足
        2. 根据 category 实例化对应 Downloader
        3. 计算日期范围（支持 auto 模式）
        4. 调用 Downloader 的对应方法
        5. 记录执行结果
        6. 更新配额消耗
        """

    def _resolve_date_range(self, task: TaskDefinition) -> tuple[str, str]:
        """解析任务的日期范围。
        - end_date="auto" 时，根据 end_offset_days 计算
        - 结合数据库中 MAX(date) 确定实际拉取范围
        """

    def _get_downloader_method(self, task: TaskDefinition) -> str:
        """根据 category + subcategory 确定调用 Downloader 的哪个方法。

        映射规则：
        - meta + basic     → download_all_metadata()
        - component        → download_all_components()
        - macro            → download_all_macro()
        - index + kline    → download_all_index()
        - kline + daily    → download_daily_kline()
        - kline + weekly   → download_weekly_kline()
        - kline + monthly  → download_monthly_kline()
        - kline + minute   → download_minute_kline()
        - financial        → download_all_financial()
        - report           → download_all_reports()
        - dividend         → download_all_dividend()
        """
```

### 6.4 `SchedulerOrchestrator` — 调度编排器

**文件**: `src/scheduler/orchestrator.py`

```python
class SchedulerOrchestrator:
    """调度编排器：加载配置 → 筛选任务 → 排序 → 执行 → 记录。"""

    def __init__(self, db_path: str, config_path: Path | None = None,
                 logger: logging.Logger | None = None):
        self.db_path = db_path
        self.config = SchedulerConfig(config_path)
        self.logger = logger or logging.getLogger("baostock.scheduler")
        self.quota_mgr: QuotaManager | None = None
        self.executor: TaskExecutor | None = None

    def run(self, trigger_type: str = "startup") -> RunResult:
        """执行一次完整的调度流程。

        流程：
        1. 加载配置
        2. 检查调度器是否启用
        3. 创建 scheduler_runs 记录
        4. 初始化 QuotaManager
        5. 筛选当前可执行的任务（时间窗口、交易日等）
        6. 按 priority 排序
        7. 依次执行每个任务
        8. 处理中断/失败
        9. 更新 scheduler_runs 状态
        10. 返回 RunResult
        """

    def _should_execute_task(self, task: TaskDefinition) -> bool:
        """判断任务当前是否应该执行。

        检查项：
        1. task.enabled
        2. schedule.type 匹配当前触发方式
        3. 时间窗口是否在允许范围内
        4. 星期几是否在允许列表中
        5. 如果 trading_day_only，检查今日是否为交易日
        """

    def _filter_tasks(self) -> list[TaskDefinition]:
        """筛选当前可执行的任务列表。"""

    def _execute_task(self, task: TaskDefinition, run_id: str) -> TaskRunResult:
        """执行单个任务并记录结果。"""
```

**RunResult 数据结构**：

```python
@dataclass
class RunResult:
    run_id: str
    trigger_type: str
    started_at: str
    finished_at: str | None = None
    status: str = "running"
    task_results: list[TaskRunResult] = field(default_factory=list)
    total_requests: int = 0
    total_rows: int = 0
```

## 8. API 请求配额管理

### 8.1 配额层级

```
每日总配额 (global.daily_request_limit = 49000)
    │
    ├── 按比例分配 (share_of_daily_limit)
    │   ├── kline_daily:     60% → 29400
    │   ├── kline_weekly:    10% → 4900
    │   └── ...
    │
    └── 按固定值分配 (max_requests)
        ├── meta_basic:      50
        ├── macro_data:      200
        └── ...
```

### 8.2 配额检查时机

| 时机 | 检查内容 | 动作 |
|------|----------|------|
| 任务开始前 | 全局剩余配额 + 任务剩余配额 | 配额不足 → 跳过任务 |
| 每批下载前 | 任务剩余配额 | 配额不足 → 保存 checkpoint，结束任务 |
| 每次 API 调用后 | 全局 + 任务配额 | 触发上限 → 优雅退出 |
| 任务完成后 | 记录实际消耗 | 写入 quota_allocation |

### 8.3 配额不足时的降级策略

```python
if quota_exceeded:
    if task.retry.get("skip_on_quota_exceeded"):
        # 跳过该任务，继续执行后续任务
        mark_skipped(reason="quota_exceeded")
        continue
    else:
        # 保存 checkpoint，终止当前任务
        save_checkpoint()
        mark_interrupted(reason="quota_exceeded")
        # 继续执行后续任务（非阻塞模式）
        continue
```

## 9. 异常处理与恢复

### 9.1 异常分类

| 异常类型 | 处理方式 | 是否重试 |
|----------|----------|----------|
| 网络超时 | 指数退避重试 | 是 |
| API 会话过期 | 重新登录后重试 | 是 |
| 配额不足 | 保存 checkpoint，跳过 | 否 |
| 数据库锁定 | 等待 30s 后重试 | 是 |
| 数据解析错误 | 记录错误，跳过该条 | 否 |
| 磁盘空间不足 | 终止调度，告警 | 否 |

### 9.2 重试策略

```python
# 默认重试策略（继承 global.default_retry）
retry_policy = task.retry or global_config.default_retry

for attempt in range(retry_policy["max_retries"]):
    try:
        result = execute()
        if success:
            break
    except RetryableError as e:
        if e.error_code in retry_policy.get("retry_on_error_codes", []):
            sleep(backoff_base ** attempt)
            continue
        else:
            raise
```

```yaml
      share_of_daily_limit: 0.5
  # 财务数据分配较大配额（全量拉取）
  - name: financial_data
    download_params:
      current_quarter_only: false  # 全量拉取
    quota:
      share_of_daily_limit: 0.3
```

## 附录 B：调度状态查询 SQL 汇总

```sql
-- 1. 查看所有任务定义
SELECT task_name, category, enabled, priority, schedule_type
FROM scheduler_tasks
ORDER BY priority;

-- 2. 查看最近 7 天运行记录
SELECT run_id, trigger_type, started_at, finished_at, status,
       total_tasks, completed_tasks, failed_tasks
FROM scheduler_runs
WHERE started_at >= date('now', '-7 days')
ORDER BY started_at DESC;

-- 3. 查看今日配额使用
SELECT task_name, allocated, consumed, remaining,
       ROUND(100.0 * consumed / NULLIF(allocated, 0), 1) as usage_pct
FROM quota_allocation
WHERE date = date('now')
ORDER BY consumed DESC;

-- 4. 查看失败最多的任务
SELECT task_name, COUNT(*) as fail_count
FROM scheduler_task_runs
WHERE status = 'failed'
GROUP BY task_name
ORDER BY fail_count DESC
LIMIT 10;

-- 5. 查看平均执行时长
SELECT task_name,
       ROUND(AVG(duration_seconds), 1) as avg_seconds,
       ROUND(MIN(duration_seconds), 1) as min_seconds,
       ROUND(MAX(duration_seconds), 1) as max_seconds
FROM scheduler_task_runs
WHERE status = 'completed'
GROUP BY task_name;
```
