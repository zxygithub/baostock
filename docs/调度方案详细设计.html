1. 设计目标

1.1 核心诉求

目标说明
可配置基于数据库表 / 数据种类进行任务配置，无需修改代码
可量化支持时间范围、拉取数据量、请求次数等维度的精确控制
可复用程序每次启动时自动读取调度配置，按配置执行拉取
可观测每次调度运行有完整的执行记录和统计
可恢复中断后可从断点恢复，不重复拉取已完成任务

1.2 设计原则

  • 向后兼容：现有 config.yaml 和下载脚本无需修改即可继续工作
  • 渐进式接入：调度器作为可选层，不强制替代现有下载流程
  • 最小侵入：复用现有 Downloader 体系，不重写下载逻辑
  • 单一职责：调度器只负责任务编排，不关心具体下载实现

2. 现状分析

2.1 当前下载流程

start.sh full/update ↓ scripts/download_all.py / update_daily.py ↓ 按固定顺序调用各 Downloader ↓ 每个 Downloader 独立管理自己的下载逻辑

2.2 存在的问题

问题影响
下载顺序硬编码调整顺序需修改 Python 代码
请求配额全局共享无法为不同数据种类分配不同配额
无任务级状态追踪不知道哪些表已拉完、哪些失败
全量/增量两套脚本维护成本高，逻辑有重复
配置分散config.yaml 控制开关和日期，代码中控制顺序和参数

2.3 现有可复用组件

组件可复用点
BaseDownloader登录、重试、checkpoint、请求计数
config_loader.pyYAML 配置加载机制
db_manager.pySQLite 连接管理、表结构定义
各具体 Downloader实际下载逻辑（无需修改）
request_count 表已有的 API 请求计数基础

4. 数据库设计

4.1 新增表清单

序号表名说明
1scheduler_tasks调度任务定义（配置持久化）
2scheduler_runs调度运行记录（每次执行的历史）
3scheduler_task_runs任务级执行记录（每个任务的执行结果）
4quota_allocation配额分配与消耗记录

4.2 表结构

4.2.1 scheduler_tasks — 调度任务定义

CREATE TABLE IF NOT EXISTS scheduler_tasks (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    task_name       TEXT    NOT NULL UNIQUE,
    data_category   TEXT    NOT NULL,
    data_subcategory TEXT,
    target_tables   TEXT    NOT NULL,
    enabled         INTEGER NOT NULL DEFAULT 1,
    priority        INTEGER NOT NULL DEFAULT 10,
    schedule_type   TEXT    NOT NULL DEFAULT 'on_startup',
    schedule_config TEXT,
    download_params TEXT,
    quota_config    TEXT,
    retry_policy    TEXT,
    created_at      TEXT    NOT NULL,
    updated_at      TEXT    NOT NULL
);

4.2.2 scheduler_runs — 调度运行记录

CREATE TABLE IF NOT EXISTS scheduler_runs (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id          TEXT    NOT NULL UNIQUE,
    trigger_type    TEXT    NOT NULL,
    started_at      TEXT    NOT NULL,
    finished_at     TEXT,
    status          TEXT    NOT NULL DEFAULT 'running',
    total_tasks     INTEGER NOT NULL DEFAULT 0,
    completed_tasks INTEGER NOT NULL DEFAULT 0,
    failed_tasks    INTEGER NOT NULL DEFAULT 0,
    skipped_tasks   INTEGER NOT NULL DEFAULT 0,
    total_requests  INTEGER NOT NULL DEFAULT 0,
    total_rows      INTEGER NOT NULL DEFAULT 0,
    error_message   TEXT,
    config_snapshot TEXT
);

4.2.3 scheduler_task_runs — 任务级执行记录

CREATE TABLE IF NOT EXISTS scheduler_task_runs (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id          TEXT    NOT NULL,
    task_name       TEXT    NOT NULL,
    started_at      TEXT    NOT NULL,
    finished_at     TEXT,
    status          TEXT    NOT NULL DEFAULT 'running',
    requests_used   INTEGER NOT NULL DEFAULT 0,
    rows_fetched    INTEGER NOT NULL DEFAULT 0,
    tables_updated  TEXT,
    error_message   TEXT,
    checkpoint      TEXT,
    duration_seconds REAL
);

CREATE INDEX idx_task_runs_run_id ON scheduler_task_runs(run_id);
CREATE INDEX idx_task_runs_task_name ON scheduler_task_runs(task_name);

4.2.4 quota_allocation — 配额分配与消耗

CREATE TABLE IF NOT EXISTS quota_allocation (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    date            TEXT    NOT NULL,
    task_name       TEXT    NOT NULL,
    allocated       INTEGER NOT NULL DEFAULT 0,
    consumed        INTEGER NOT NULL DEFAULT 0,
    remaining       INTEGER NOT NULL DEFAULT 0,
    updated_at      TEXT,
    UNIQUE(date, task_name)
);

CREATE INDEX idx_quota_date ON quota_allocation(date);

5. 调度配置模型

5.1 scheduler.yaml 文件格式

# 全局设置
global:
  daily_request_limit: 49000
  enabled: true
  default_retry:
    max_retries: 3
    backoff_base: 2

# 任务定义列表（按 priority 从小到大排序执行）
tasks:
  - name: meta_basic
    category: meta
    subcategory: basic
    tables: ["stock_basic", "trade_dates", "stock_industry"]
    enabled: true
    priority: 1
    schedule:
      type: on_startup
      trading_day_only: false
    download_params:
      full_refresh: true
    quota:
      max_requests: 50
      share_of_daily_limit: 0.001

  - name: kline_daily
    category: kline
    subcategory: daily
    tables: ["all_stock_daily"]
    enabled: true
    priority: 5
    schedule:
      type: daily
      trading_day_only: true
      time_window:
        start: "01:00"
        end: "06:00"
      max_duration_minutes: 180
    download_params:
      start_date: "1990-12-19"
      end_date: auto
      end_offset_days: -1
      adjustflags: [1, 2, 3]
      batch_size: 200
      batch_sleep: 2
    quota:
      max_requests: 30000
      share_of_daily_limit: 0.6

5.2 配置字段说明

任务级字段

字段类型必填默认值说明
namestring是—任务唯一标识
categorystring是—数据大类，映射到具体 Downloader
subcategorystring否null数据子类
tablesarray是—目标数据库表名列表
enabledbool否true是否启用
priorityint否10执行优先级（小→大）
scheduleobject否见下文调度策略
download_paramsobject否{}传递给 Downloader 的参数
quotaobject否{}配额限制
retryobject否继承 global重试策略

schedule 字段

字段类型说明
typestringon_startup / daily / weekly / monthly / manual
weekdaysarray允许执行的星期，[1,2,3,4,5] 表示工作日
trading_day_onlybool是否仅交易日执行
time_windowobject执行时间窗口 {start: "01:00", end: "06:00"}
max_duration_minutesint最大执行时长

6. 核心模块设计

6.1 SchedulerConfig — 配置加载器

文件: src/scheduler/config.py

  • 解析 scheduler.yaml 为结构化对象
  • 提供任务查询接口
  • 配置校验（必填字段、类型检查）
  • 配置缓存（避免重复读取文件）

6.2 QuotaManager — 配额管理器

文件: src/scheduler/quota.py

  • 为任务列表分配配额（固定配额 / 比例分配）
  • 检查任务是否还有可用配额
  • 消耗配额并返回结果
  • 获取任务/全局剩余配额
  • 将配额状态写入 quota_allocation 表

6.3 TaskExecutor — 任务执行器

文件: src/scheduler/executor.py

  • 维护 category → Downloader 映射表
  • 解析任务的日期范围（支持 auto 模式）
  • 根据 category + subcategory 确定调用方法
  • 实例化对应 Downloader 并执行下载
  • 记录执行结果并更新配额消耗

6.4 SchedulerOrchestrator — 调度编排器

文件: src/scheduler/orchestrator.py

  • 加载配置，获取已启用任务列表
  • 创建 scheduler_runs 记录
  • 初始化 QuotaManager 并分配配额
  • 筛选可执行任务（时间窗口、交易日等）
  • 按 priority 排序后依次执行
  • 处理中断/失败，更新运行状态

6.5 模块目录结构

src/ ├── scheduler/ │ ├── __init__.py # 包初始化，导出核心类 │ ├── config.py # SchedulerConfig + 数据类 │ ├── orchestrator.py # SchedulerOrchestrator │ ├── executor.py # TaskExecutor + CATEGORY_MAP │ ├── quota.py # QuotaManager │ └── models.py # 数据模型（dataclass 定义）

7. 执行流程

7.1 程序启动流程（调度器模式）

start.sh scheduled ↓ scripts/run_scheduler.py ↓ ┌─────────────────────────────────────────────┐ │ 1. 初始化日志 │ │ 2. 初始化数据库表（含调度表） │ │ 3. 加载 scheduler.yaml │ │ 4. 检查 global.enabled │ │ ↓ false → 回退到原有 download_all.py │ │ ↓ true → 继续 │ │ 5. 创建 SchedulerOrchestrator │ │ 6. orchestrator.run(trigger_type="startup") │ │ ↓ │ │ a. 加载配置，获取已启用任务列表 │ │ b. 创建 scheduler_runs 记录 │ │ c. 初始化 QuotaManager │ │ d. 分配配额 │ │ e. 筛选可执行任务 │ │ f. 按 priority 排序 │ │ g. 依次执行每个任务: │ │ - 检查时间窗口 │ │ - 检查交易日 │ │ - 检查配额 │ │ - 实例化 Downloader │ │ - 执行下载 │ │ - 记录结果 │ │ h. 更新 scheduler_runs 状态 │ │ i. 持久化配额状态 │ │ 7. 输出运行摘要 │ └─────────────────────────────────────────────┘

7.2 任务执行详细流程

执行任务 "kline_daily" ↓ 1. 检查任务是否应该执行 - enabled? ✓ - schedule.type 匹配? ✓ - 时间窗口 [01:00, 06:00]? 当前 02:30 ✓ - 交易日? ✓ 2. 检查配额 - allocated: 30000 - consumed: 0 - remaining: 30000 ✓ 3. 创建 scheduler_task_runs 记录 (status = "running") 4. 解析日期范围 - start_date: "1990-12-19" - end_date: auto → 计算为 "2026-05-15" - DB max_date: "2026-05-14" - 实际拉取: "2026-05-15" → "2026-05-15" 5. 实例化 KlineDownloader 6. 执行 download_daily_kline() - 获取股票列表 - 分批处理（batch_size=200） - 每批检查配额是否充足 7. 记录执行结果 - status: "completed" - requests_used: 15234 - rows_fetched: 2345678 - duration_seconds: 3456.7 8. 更新配额消耗 (consumed += 15234) 9. 提交 scheduler_task_runs

7.3 中断恢复流程

程序收到 SIGINT / SIGTERM ↓ 1. 捕获信号 2. 当前任务保存 checkpoint 3. 更新 scheduler_task_runs: status = "interrupted" 4. 更新 scheduler_runs: status = "interrupted" 5. 优雅退出 下次启动时: 1. 读取 scheduler_task_runs 2. 找到 interrupted 状态的任务 3. 从 checkpoint 恢复 4. 继续执行

8. API 请求配额管理

8.1 配额层级

每日总配额 (global.daily_request_limit = 49000) │ ├── 按比例分配 (share_of_daily_limit) │ ├── kline_daily: 60% → 29400 │ ├── kline_weekly: 10% → 4900 │ └── ... │ └── 按固定值分配 (max_requests) ├── meta_basic: 50 ├── macro_data: 200 └── ...

8.2 配额检查时机

时机检查内容动作
任务开始前全局剩余配额 + 任务剩余配额配额不足 → 跳过任务
每批下载前任务剩余配额配额不足 → 保存 checkpoint，结束任务
每次 API 调用后全局 + 任务配额触发上限 → 优雅退出
任务完成后记录实际消耗写入 quota_allocation

8.3 配额不足时的降级策略

if quota_exceeded:
    if task.retry.get("skip_on_quota_exceeded"):
        # 跳过该任务，继续执行后续任务
        mark_skipped(reason="quota_exceeded")
        continue
    else:
        # 保存 checkpoint，终止当前任务
        save_checkpoint()
        mark_interrupted(reason="quota_exceeded")
        # 继续执行后续任务（非阻塞模式）
        continue

12. 使用示例

12.1 基本使用

# 1. 生成初始调度配置
.venv/bin/python scripts/migrate_to_scheduler.py

# 2. 编辑 scheduler.yaml，调整任务参数
vim scheduler.yaml

# 3. 运行调度器
./start.sh scheduled

# 4. 查看运行记录
.venv/bin/python scripts/query_scheduler_status.py

12.2 查询调度状态（SQL）

-- 查看最近一次调度运行
SELECT * FROM scheduler_runs ORDER BY started_at DESC LIMIT 1;

-- 查看各任务执行情况
SELECT task_name, status, requests_used, rows_fetched, duration_seconds
FROM scheduler_task_runs
WHERE run_id = (SELECT run_id FROM scheduler_runs ORDER BY started_at DESC LIMIT 1)
ORDER BY priority;

-- 查看配额使用情况
SELECT task_name, allocated, consumed, remaining
FROM quota_allocation
WHERE date = date('now');

12.3 典型场景配置

场景 A：仅更新 K 线日线

将 kline_daily 的 share_of_daily_limit 设为 0.9，其他任务禁用。

场景 B：全量数据初始化

提高 daily_request_limit 至 95000，所有任务启用，财务数据设置 current_quarter_only: false。

场景 C：仅工作日凌晨更新

设置 weekdays: [1,2,3,4,5] + trading_day_only: true + time_window: 01:00-06:00。

附录 B：调度状态查询 SQL 汇总

-- 1. 查看所有任务定义
SELECT task_name, category, enabled, priority, schedule_type
FROM scheduler_tasks
ORDER BY priority;

-- 2. 查看最近 7 天运行记录
SELECT run_id, trigger_type, started_at, finished_at, status,
       total_tasks, completed_tasks, failed_tasks
FROM scheduler_runs
WHERE started_at >= date('now', '-7 days')
ORDER BY started_at DESC;

-- 3. 查看今日配额使用
SELECT task_name, allocated, consumed, remaining,
       ROUND(100.0 * consumed / NULLIF(allocated, 0), 1) as usage_pct
FROM quota_allocation
WHERE date = date('now')
ORDER BY consumed DESC;

-- 4. 查看失败最多的任务
SELECT task_name, COUNT(*) as fail_count
FROM scheduler_task_runs
WHERE status = 'failed'
GROUP BY task_name
ORDER BY fail_count DESC
LIMIT 10;

-- 5. 查看平均执行时长
SELECT task_name,
       ROUND(AVG(duration_seconds), 1) as avg_seconds,
       ROUND(MIN(duration_seconds), 1) as min_seconds,
       ROUND(MAX(duration_seconds), 1) as max_seconds
FROM scheduler_task_runs
WHERE status = 'completed'
GROUP BY task_name;
