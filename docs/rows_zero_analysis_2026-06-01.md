```
   78 次 = 历史季度（2025-2026 Q1 等，极少数股票缺失）
```

- 全量下载遍历 2007→2026 所有季度
- `_find_missing_quarters` 跳过了 255,988 条已有记录
- 剩余 5,292 条待下载任务中，**5,214 条全是 2026 Q2**
- Q2 财报 7-8 月才披露，**全市场无数据**

## 四、根本原因

**cron 每天 00:05 运行 `start.sh full`（全量下载），而非增量更新。**

全量下载的财务数据流程：
1. 遍历 2007→2026 所有季度 × 所有股票
2. 用 `_find_missing_quarters` 通过 LEFT JOIN 过滤已有记录
3. 对缺失的 `(code, year, quarter)` 逐一发起 API 请求
4. **Q2 2026 全市场未发布** → 所有股票的 Q2 查询都返回 rows=0
5. growth_data 历史缺口大（166,735 条）→ 每天消耗大量 API 额度填充无效查询
6. **API 额度在 08:51 耗尽**（48,999/49,000），程序提前退出，后续数据未下载

---

## 五、建议

### 🔴 高优先级（立即可做，每天节省 ~9K 次无效请求）

#### 1. 跳过未发布的当前季度

在 `financial_downloader.py` 的 `_download_quarterly_data` 中，当 `year == current_year and quarter == current_quarter` 时，财报尚未发布，应直接跳过：

```python
# 跳过当前季度（财报尚未发布）
if year == current_year and quarter == current_quarter:
    continue
```

**效果**：直接消除 ~3,700 次 rows=0（2601+5214+558 中的 Q2 部分），节省约 7.5% 的 API 额度。

#### 2. 全量下载改为增量更新

cron 应将 `./start.sh full` 改为 `./start.sh update`：

- `update_daily.py` 只更新最近 1-2 个季度的财务数据
- 不会遍历 2007-2026 全部历史
- 已有数据不会被重复检查
- 预计每天 API 请求从 ~49K 降至 ~16K

### 🟡 中优先级

#### 3. 缓存空结果（负缓存）

对 `rows=0` 的 `(code, year, quarter)` 组合写入数据库标记，下次 `_find_missing_quarters` 时排除。

目前代码对非近年空结果会写入占位记录（第 91-97 行），但**近年空结果（Q2 2026）不写入**，导致每天重复查询。

修改方案：对所有 `rows=0` 都写入占位记录，而非仅 `year not in recent_years`。

#### 4. 财务数据下载频率降低

财务数据按季度发布，不需要每天检查。可以：
- 全量下载中跳过财务阶段（`--skip-financial`）
- 单独设置 cron 任务，每周或每季度运行一次财务下载

### 🟢 低优先级

#### 5. 科创板 IPO 日期过滤

growth_data 中 286 次 2019 年空请求来自 688xxx 科创板股票。`get_stock_years` 已基于 IPO 日期过滤，但 growth_data 的 API 本身对该类股票返回空——可能是 BaoStock 的 growth_data 对该类股票数据不完整。可以进一步验证 IPO 过滤是否生效。
